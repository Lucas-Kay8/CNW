## 第5页：到底什么是 Agent？

## 第12页：内部结构大解剖（一）：多角色协作

- **幻灯片标题**：Agent 内部，其实是一个“专家团队”

	- **角色 2：分析师 (Analyst)**
		- **职责**：负责**深度阅读和理解**原始报告，提取关键信息和数据。
	- **角色 3：内容架构师 (Architect)**
		- **职责**：根据分析师提取的要点，**设计课件的逻辑结构**（如总分总、层层递进）。
	- **角色 4：文案写手 (Writer)**
		- **职责**：用通俗易懂的语言，**撰写每一页 PPT 的具体内容**。
- **讲师旁白**：“就像一个项目团队，有做计划的，有做分析的，有搭框架的，有写稿的。Agent 内部也是这样通过角色协同，来完成复杂任务的。”

## 第13页：内部结构大解剖（二）：触发与数据流

- **幻灯片标题**：数据是如何在 Agent 内部“奔跑”的？
- **要点**：

		- **Step 1**: 报告 → **分析师** → 提取出“核心观点摘要 (JSON)”。
		- **Step 2**: “核心观点摘要” → **内容架构师** → 设计出“课件结构树 (JSON)”。

	- 每一步的**输出**，都成为下一步的**输入**。
- **讲师旁白**：“大家看，这个过程就像一条流水线。原始的报告被一步步加工，最后变成了我们想要的 PPT 大纲。每一步之间通过结构化的数据（JSON）来传递信息。”

## 第15页：内部结构大解剖（四）：评估与迭代

## 第17页：Agent 的“思考”过程：ReAct 框架简介

- **幻灯片标题**：揭秘 Agent 的思考过程：ReAct 框架
- **要点**：
	- Agent 如何决定“下一步该做什么”？—— 核心在于 **ReAct (Reason + Act)** 框架。
	- **Reason (思考)**：Agent **在脑中分析**当前情况，我有什么信息？我的目标是什么？我应该用哪个工具？
	- **Act (行动)**：Agent **选择并执行**一个动作（如：调用`web_search`工具搜索“什么是AIGC”）。
	- **Observe (观察)**：Agent **接收行动的结果**（如：搜索引擎返回的结果）。
	- 然后，Agent 带着新的“观察”，进入下一次“思考-行动-观察”循环，直到任务完成。
- **讲师旁白**：“这个 ReAct 循环，就像我们人类解决问题一样：想一想，做一步，看一看结果，再想下一步。Agent 就是通过这个机制，一步步逼近最终目标的。”
- **视觉建议**：一个简洁的循环图，清晰地展示 “Reason -> Act -> Observe” 三个阶段，并循环回 Reason。

- **讲师旁白**：“理论讲完了，我们来点实际的。这部分不要求大家现场操作，而是带大家‘预览’一下，在一个真实的 Agent 平台上，我们之前讨论的那些概念，都对应着哪些按钮和操作。”
- **视觉建议**：一张图片，左边是上一模块的“设计蓝图”，右边是一个模糊的、通用的平台界面截图，中间有一个巨大的箭头，表示“转化/实现”。

## 第20页：主流平台的核心三要素

- **幻灯片标题**：万变不离其宗：所有 Agent 平台的“三件套”
- **要点**：
	- 无论界面如何变化，核心都是这三大组成部分：
	- **1. 大语言模型 (LLM) - 大脑**
		- **平台操作**：通常是一个**下拉菜单**，让你选择要使用的“大脑”。
		- **你的选择**：是选更强大但更贵的 GPT-4，还是更快速便宜的 Llama 3？这取决于你的**任务复杂度和成本预算**。
	- **2. 知识库 (Knowledge) - 记忆**
		- **平台操作**：一个**“上传”或“连接”**的区域。
		- **你的选择**：是上传几篇 PDF 文档，还是连接一个在线的 Notion 数据库？
	- **3. 插件/工具 (Plugins/Tools) - 手脚**
		- **平台操作**：一个**“工具商店”或“能力市场”**。
		- **你的选择**：是添加“网页搜索”工具，还是“计算器”工具，或是“调用API”的工具？
- **讲师旁白**：“记住这张图。你未来接触任何新平台，第一件事就是找这‘三件套’分别在哪里。找到了，你就掌握了 80% 的用法。”
- **视觉建议**：一个简洁的平台界面示意图，用三个高亮框分别圈出“模型选择”、“知识库管理”、“工具市场”三个区域。

## 第21页：Agent 开发的完整生命周期

- **幻灯片标题**：创造一个 Agent，总共分几步？
- **要点**：
	- 一个完整的 Agent 项目，遵循一个清晰的开发路径：
	- **Phase 1: 需求思考** - 我想让 Agent **解决什么问题**？
	- **Phase 2: 流程设计** - 参考我们的“设计蓝图”，**规划角色、数据流和工具**。
	- **Phase 3: 编排搭建** - 在平台上**拖拉拽、点选配置**，实现你的设计。
	- **Phase 4: 测试与迭代** - **试用、发现问题**，并返回上一步进行调整。
	- **Phase 5: 发布与运行** - 将你的 Agent **分享给他人**或集成到其他应用中。
	- **Phase 6: 复盘与优化** - 收集用户反馈，**持续改进** Agent 的表现。
- **讲师旁白**：“这六步，和我们开发任何一个产品的思路都是一致的。AI 应用开发，并不是什么神秘的魔法，而是一套严谨的工程方法。”
- **视觉建议**：一个从左到右的六阶段流程图，每个阶段配一个关键词和一个小图标。

## 第22页：实战预览（一）：定义目标与角色

- **幻灯片标题**：第一步：给你的 Agent 一个清晰的“人设”
- **要点**：
	- **平台操作**：进入 Agent 创建页面，首先会看到一个巨大的**“指令 (Instruction)”或“提示词 (Prompt)”输入框**。
	- **你要做什么**：在这里写下对 Agent 的**总体要求**，也就是我们之前定义的“角色团队”和“最终目标”。
	- **示例**：
	> “你是一个专业的培训课件设计师。你的任务是读取用户上传的行业报告，为非技术背景的大学生制作一份20页左右的培训PPT大纲。你需要先分析报告，然后设计结构，最后用通俗的语言撰写内容。”
- **讲师旁白**：“这一步，就是在为你的 Agent 注入灵魂。你希望它扮演什么角色，完成什么使命，都要在这里定义清楚。这是所有后续工作的基础。”
- **视觉建议**：一个平台界面截图，高亮显示“Prompt”输入框，并填入示例文字。

## 第23页：实战预览（二）：配置知识与工具

- **幻灯片标题**：第二步：为你的 Agent 连接“记忆”和“手脚”
- **要点**：
	- **配置知识库**：
		- **平台操作**：找到“知识库”或“Knowledge”区域，点击**“上传文件”**。
		- **你要做什么**：将我们的“行业研究报告.pdf”上传上去。
	- **配置工具**：
		- **平台操作**：找到“工具”或“Tools”区域，点击**“添加”**。
		- **你要做什么**：从工具市场中，**勾选**`Web Search`(网页搜索) 和 `Code Interpreter`(代码解释器，常用于数据分析和文件处理)。
- **讲师旁白**：“现在，我们把‘原材料’（知识库）和‘工具箱’（工具）都交给 Agent。它的大脑、记忆、手脚就都齐全了。”
- **视觉建议**：界面截图，左侧高亮“上传文件”按钮，右侧高亮“添加工具”并展示勾选了几个工具的弹窗。

## 第24页：实战预览（三）：编排与测试

- **幻灯片标题**：第三步：像搭积木一样，构建工作流
- **要点**：
	- 对于更复杂的 Agent，你需要一个**可视化的工作流编辑器**。
	- **平台操作**：
		- **拖拽节点**：从左侧的节点库中，拖拽出“LLM调用”、“读取文件”、“代码执行”等节点到画布上。
		- **连接节点**：用鼠标**将节点连接起来**，定义我们之前设计的“数据流”。
		- **配置节点**：点击每个节点，在右侧的配置面板中**填写详细的 Prompt 或参数**。
	- **测试**：
		- 点击界面右上角的**“运行”**按钮，观察 Agent 是否按照你的设计执行。
		- 在**日志 (Logs)**区域，查看每一步的输入和输出，定位问题。
- **讲师旁白**：“这一步最有趣，就像在搭乐高。你把代表不同能力的模块组合起来，定义它们之间的连接关系。如果运行出错了，别慌，日志会告诉你问题出在哪一步。”
- **视觉建议**：一个可视化的工作流编辑器界面示意图，包含几个被箭头连接起来的方框（节点），右侧有一个配置面板。
